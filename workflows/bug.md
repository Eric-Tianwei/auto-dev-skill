# Bug 工作流

严格推理、切实验证。修复必须有失败测试先行。

## 步骤

1. **理解报告**：读任务条目的 `symptom` 和 `repro`（复现步骤）。信息不足 → 写 NEEDS_REVIEW 停下。
2. **建分支**：`fix/<task-id>`。
3. **最小复现 + 失败测试**（强制）：
   - 派 subagent 产出：最小复现脚本 / 红色单元测试 / 日志证据。
   - 主循环跑一遍确认测试确实失败，且失败原因与报告一致。
   - 复现不了 → 停下写 NEEDS_REVIEW，不要凭猜测修。
4. **定位根因**：读代码、追调用链。**不要**只让症状消失（例：吞异常、加 try/except pass、放宽断言）。
5. **修复**：最小改动。改动范围超出预期 → 停下向用户确认。
6. **验证**：
   - 第 3 步的失败测试现在通过。
   - 跑全量测试，确认没有其他用例从绿变红。
   - 如果项目有集成测试/端到端测试且与改动相关，也要跑。
7. **独立 review**：commit 前派一个 subagent 独立审查 diff，重点问三个问题：
   - 这个修复是否触到根因？
   - 有没有引入回归风险？
   - 有没有用"绕过"代替"修复"（忽略异常、改测试期望、加特殊分支）？
   - review 若提出实质问题 → 处理完再 commit。
8. **Commit**：commit message 必须包含根因一句话 + 关联任务 id。
9. **回主循环**：JOURNAL 记录分支名 + 根因 + 影响范围，backlog 标 `done`。

## 硬性禁止

- 不得为了让测试通过而修改/删除原有断言。
- 不得用 `try/except pass`、`// ignore`、`@ts-ignore` 等掩盖问题，除非用户显式要求。
- 不得跳过测试（`.skip`、`xfail`、注释掉）。确需跳过 → 停下写 NEEDS_REVIEW。
- 不得用 `--no-verify` 跳过 hook。
