# Feature 工作流

鼓励探索。允许产出多个方案分支供用户挑选。

## 步骤

1. **理解需求**：读任务条目的 `goal` 和 `acceptance`。不清楚 → 写 NEEDS_REVIEW 停下，不要猜。
2. **决定方案数**：
   - 需求清晰、方向明确 → 1 个分支。
   - 有明显设计权衡（架构、库选型、交互方式） → 2–3 个变体并行。
3. **建分支**：从当前 `main`（或任务指定 base）切 `feat/<task-id>[-vN]`。
4. **并行变体**（如选择多方案）：派 subagent + `isolation: worktree`，每个 agent 的 prompt 必须包含：
   - 任务 goal 与 acceptance
   - 该变体要探索的设计方向（一句话）
   - 要求回传：分支名、`DESIGN.md` 路径、自测结果
5. **实现 + 自测**：跑项目已有的测试/类型检查/lint。没有测试框架 → 至少手动验收一次主路径。
6. **写设计备忘**：每个变体分支根目录放一份 `DESIGN-<task-id>.md`，≤30 行，包含：动机、方案要点、权衡、放弃的路。
7. **Commit**：遵循仓库 commit 风格。**不要** push，**不要** 合并到 main。
8. **回主循环**：JOURNAL 记录分支名列表+一句话结论，backlog 该条标 `done-pending-review`（等用户挑选）。

## 注意

- 不要跨任务改动无关代码。看到顺手想清理的东西 → 新建一条 backlog 任务，不要夹带。
- 不要为"未来可能"加抽象层。三行重复好过一个过早的基类。
- Feature 类失败 1 次可以换思路再试 1 次；连续 2 次仍失败 → 停下写 NEEDS_REVIEW。
